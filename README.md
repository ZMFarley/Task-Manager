# Task Manager API Documentation

## Overview

This is my CIS-470: Software Testing & Automation final project. It a FastAPI-based task manager backed by a SQLite database. The application exposes REST endpoints for creating, retrieving, updating, and deleting task records. When the application starts, it initializes the database from `db.sql` if the table does not already exist. The purpose of this project is to provide a full stack application that can be used and deployed via a CI/CD Workflow to github, running the included test suite on push or pull requests. From there, the application is hosted as a web service on Render, which continously checks for updates, satisfying the requirements of CI/CD.

The API also redirects the root route (`/`) to the Swagger UI at `/docs`, which provides an interactive interface for testing the endpoints.

## Setup and Run

Install the dependencies:

```bash
pip install -r requirements.txt
```

Start the API:

```bash
uvicorn api:app --reload
```

Open the interactive API documentation at:

```text
http://localhost:8000/docs
```

## Data Model and Schema

### Database Schema

The application uses one SQLite table: `Tasks`.

| Column | Type | Constraints | Description |
|--------|------|-------------|-------------|
| `id` | `INTEGER` | `PRIMARY KEY`, `NOT NULL` | Unique identifier for each task |
| `due_date` | `DATE` | `NOT NULL` | Due date for the task in `YYYY-MM-DD` format |
| `task` | `TEXT` | Application-enforced length validation | Text description of the task |

Schema definition from `db.sql`:

```sql
CREATE TABLE IF NOT EXISTS Tasks(
    id INTEGER NOT NULL PRIMARY KEY,
    due_date DATE NOT NULL,
    task TEXT
);
```

### Entity: Task

`Task` is the core and only business entity in the software.

| Attribute | Type | Required | Validation | Description |
|-----------|------|----------|------------|-------------|
| `id` | `int` | Generated by database | Must be greater than `0` when supplied in routes | Unique task identifier |
| `due_date` | `date` | Yes for create and full update | Must be a valid date in `YYYY-MM-DD` format | Date the task is due |
| `task` | `str` | Yes for create and full update | Minimum length `1`, maximum length `255` |  task description |

### API Models

Task Manager defines three Pydantic models that represent the same underlying task entity in different request/response contexts.

#### `Task`

Used for `POST /tasks` and `PUT /tasks/{id}`.

| Field | Type | Required | Validation |
|-------|------|----------|------------|
| `due_date` | `date` | Yes | Valid `YYYY-MM-DD` date |
| `task` | `str` | Yes | `1-255` characters |

#### `TaskGrabber`

Used as the response model for successful `POST`, `GET`, `PATCH`, and `PUT` requests.

| Field | Type | Required | Validation |
|-------|------|----------|------------|
| `id` | `int` | Yes | Positive integer |
| `due_date` | `date` | Yes | Valid `YYYY-MM-DD` date |
| `task` | `str` | Yes | `1-255` characters |

#### `TaskPatch`

Used for partial updates in `PATCH /tasks/{id}`.

| Field | Type | Required | Validation |
|-------|------|----------|------------|
| `due_date` | `date` | No | Valid `YYYY-MM-DD` date if present |
| `task` | `str` | No | `1-255` characters if present |

At least one field must be present in a PATCH request. If the request body is empty, the API returns `400 Bad Request` with `No Data provided`.

### Relationships

This system has a very simple data model:

- There is one entity: `Task`.
- Each row in the `Tasks` table represents one independent task.
- There are no foreign keys, join tables, parent-child records, or many-to-many relationships.
- The API routes operate directly on individual `Task` records using the `id` field.

Because the project only manages standalone tasks, the main relationship in the software is between:

- The `Tasks` database table
- The Pydantic models (`Task`, `TaskGrabber`, `TaskPatch`)
- The REST endpoints that create, read, update, and delete those records

## Usage

### How the Software Is Used

The software is used as a REST API for task management. A client sends HTTP requests to create tasks, retrieve a task by ID, modify an existing task, or delete a task. The Swagger UI generated by FastAPI was choosen as an expedient method to use the application manually during development.

### API Endpoints

| Method | Route | Purpose | Request Body | Success Response |
|--------|-------|---------|--------------|------------------|
| `GET` | `/` | Redirects to Swagger UI | None | Redirect to `/docs` |
| `POST` | `/tasks` | Create a new task | `Task` | `201 Created` with `TaskGrabber` |
| `GET` | `/tasks/{id}` | Retrieve a task by ID | None | `200 OK` with `TaskGrabber` |
| `PATCH` | `/tasks/{id}` | Partially update a task | `TaskPatch` | `200 OK` with `TaskGrabber` |
| `PUT` | `/tasks/{id}` | Fully replace a task | `Task` | `200 OK` with `TaskGrabber` |
| `DELETE` | `/tasks/{id}` | Delete a task by ID | None | `204 No Content` |

### Usage Flow

1. A user creates a task by sending a `POST /tasks` request with a due date and task description.
2. The application inserts that task into the `Tasks` table and returns the created record, including its generated `id`.
3. The user can retrieve the task later with `GET /tasks/{id}`.
4. The user can partially edit the task with `PATCH /tasks/{id}` or fully replace it with `PUT /tasks/{id}`.
5. The user can remove the task with `DELETE /tasks/{id}` when it is no longer needed.

### Example Requests

Create a task:

```json
{
  "due_date": "2026-05-01",
  "task": "Finish documentation"
}
```

Example response:

```json
{
  "id": 1,
  "due_date": "2026-05-01",
  "task": "Finish  documentation"
}
```

Partial update:

```json
{
  "task": "Finish data model and usage sections"
}
```

Full update:

```json
{
  "due_date": "2026-05-03",
  "task": "Submit final Task Manager documentation"
}
```

### Validation Rules

The API enforces the following rules through FastAPI and Pydantic validation, which are also covered in the black-box and white-box tests:

- `id` must be an integer greater than `0` and no larger than `9223372036854775807`
- `task` must contain between `1` and `255` characters when required or provided
- `due_date` must be a valid date in `YYYY-MM-DD` format
- `POST` and `PUT` require both `due_date` and `task`
- `PATCH` allows either field, but not an empty request body
- Requests for non-existent task IDs return `404 Task Not Found`
